# 项目规则

## 规则一：代码权限
**没有用户的明确许可，不得写任何代码。** 所有代码的编写必须经过用户口头或书面同意。

## 规则二：MVP 优先
每个功能以"最小可用产品"为标准实现，先跑通再优化。不提前设计、不过度工程。

## 规则三：先商量后动手
任何架构决策、技术选型、功能规划必须先和用户讨论，达成一致后再执行。

## 项目架构原则
模块之间严格解耦，每个模块只负责一个职责，不和其他模块冗杂：
- `llm` — 只管模型通信
- `stm` — 只管短期记忆（对话上下文）
- `ltm` — 只管长期记忆（持久化存储）
- `tool` — 只管工具注册与执行
- `agent` — 只管调度决策
- `ui` — 只管用户界面
- 依赖方向：`ui → agent → llm / tool / stm / ltm`，单向调用，禁止反向依赖。

## 规则四：协作式编码
用户写核心逻辑（LLM 调用、关键算法等），AI 负责：
1. **写之前引导**：告诉用户要写什么、用什么函数/包、代码怎么组织
2. **写之后补齐**：补异常处理、消息转换、胶水代码、配置读取、UI 渲染等周边代码
3. **不假设用户什么都会**：明确提示所需的函数名、方法、导入语句

## 规则五：Git 操作权限
**没有用户的特别允许，不得主动推送 GitHub。** 仓库地址：`https://github.com/xzd62/767676.git`
